# 术语统一规范

> 文档编号：00
> 更新日期：2026-03-04
> 适用范围：本项目所有设计文档、代码、API、配置及沟通语境

## 1. 目标

统一权限模型相关术语的定义、英文标识、中文译名及使用约束，确保：

1. 设计文档、代码实现、API 接口、配置文件之间的术语一致性。
2. 跨团队沟通时语义无歧义。
3. 新成员可快速理解核心概念。

## 2. 核心六元组

| 中文术语     | 英文标识    | 定义                                                             | 使用约束                                        |
| ------------ | ----------- | ---------------------------------------------------------------- | ----------------------------------------------- |
| **主体**     | `Subject`   | 发起行为或被治理的实体，包括自然人、组织、职责、机器账号、代理等 | 禁止用"用户"泛指，需区分具体类型                |
| **客体**     | `Object`    | 被访问或被管理的目标资源，包括业务资产、治理资产、组合资产等     | 禁止用"资源"混淆，`Resource` 仅用于外部标准映射 |
| **动作**     | `Action`    | 主体请求对客体执行的操作，如 `read`、`update`、`grant` 等        | 动作必须在 `action_catalog` 中注册              |
| **关系**     | `Relation`  | 节点之间（主体-主体、客体-客体、主体-客体）的连接边              | 必须指定 `relation_type`，禁止隐式关系          |
| **上下文**   | `Context`   | 鉴权请求的环境条件，包括时间、地点、终端、会话、风险标签等       | 不可与 `Conditions` 混用，后者是策略条件表达式  |
| **生命周期** | `Lifecycle` | 实体从创建到消亡的状态变迁，以及事件驱动的规则更新机制           | 必须与"有效期"区分，`Validity` 是时间区间       |

## 3. 数据单元术语

| 中文术语         | 英文标识         | 定义                     | 核心属性                                                                                            |
| ---------------- | ---------------- | ------------------------ | --------------------------------------------------------------------------------------------------- |
| **主体节点**     | `SubjectNode`    | 主体的数据表示单元       | `id`、`type`、`state`、`labels`                                                                     |
| **客体节点**     | `ObjectNode`     | 客体的数据表示单元       | `id`、`type`、`sensitivity`、`owner_ref`、`labels`                                                  |
| **关系边**       | `RelationEdge`   | 节点之间的连接关系       | `from`、`to`、`relation_type`、`scope`、`validity`、`source`                                        |
| **原子动作**     | `ActionAtom`     | 最细粒度的操作能力       | `action_id`、`action_domain`、`risk_level`                                                          |
| **策略规则**     | `PolicyRule`     | 授权决策的最小规则单元   | `subject_selector`、`object_selector`、`action_set`、`effect`、`conditions`、`priority`、`validity` |
| **决策记录**     | `DecisionRecord` | 单次鉴权的完整可追溯记录 | `request`、`matched_rules`、`overridden_rules`、`final_effect`、`reason`                            |
| **生命周期事件** | `LifecycleEvent` | 触发关系更新的事件       | `event_type`、`target`、`occurred_at`、`operator`                                                   |

## 6. 关系类型术语

### 6.1 主体关系

| 中文术语   | 英文标识           | 语义                   | 方向               |
| ---------- | ------------------ | ---------------------- | ------------------ |
| **隶属**   | `belongs_to`       | 主体归属于某组织单元   | 用户 → 部门        |
| **层级**   | `parent_of`        | 组织单元之间的层级包含 | 事业群 → 部门      |
| **参与**   | `member_of`        | 主体参与某虚拟协作组   | 用户 → 项目组      |
| **协作**   | `collaborate_with` | 跨组织的协作关系       | 项目组 ↔ 部门      |
| **管理**   | `manages`          | 治理权限关系           | 管理员 → 部门/用户 |
| **可授权** | `can_grant`        | 具有授权能力           | 管理员 → 资产域    |
| **委派**   | `delegates_to`     | 权限或职责的转授       | 审批人 → 代理      |

### 6.2 客体关系

| 中文术语   | 英文标识      | 语义               | 权限影响         |
| ---------- | ------------- | ------------------ | ---------------- |
| **包含**   | `contains`    | 父子层级关系       | 父授权可传播至子 |
| **层级**   | `parent_of`   | 客体层级关系       | 同 `contains`    |
| **衍生**   | `derives_to`  | 从源客体创建的副本 | 可继承源约束     |
| **克隆自** | `cloned_from` | 衍生的反向关系     | 用于溯源         |
| **依赖**   | `depends_on`  | 客体间的依赖关系   | 联动检查要求     |
| **归属**   | `owned_by`    | 客体的所有者关系   | 决定默认管理权   |
| **引用**   | `references`  | 客体间的引用关系   | 不影响权限传播   |

## 7. 策略与决策术语

### 7.1 策略组件

| 中文术语       | 英文标识          | 定义                 | 取值/格式                                                         |
| -------------- | ----------------- | -------------------- | ----------------------------------------------------------------- |
| **主体选择器** | `SubjectSelector` | 筛选目标主体的表达式 | DSL 表达式，如 `subject.relations includes member_of(group:g1)`   |
| **客体选择器** | `ObjectSelector`  | 筛选目标客体的表达式 | DSL 表达式，如 `object.type == kb and object.sensitivity == high` |
| **动作集合**   | `ActionSet`       | 规则覆盖的动作列表   | 已注册动作的非空集合                                              |
| **效果**       | `Effect`          | 规则的判定结果       | `allow` 或 `deny`                                                 |
| **条件**       | `Conditions`      | 规则生效的附加条件   | DSL 表达式                                                        |
| **优先级**     | `Priority`        | 冲突裁决时的排序权重 | 整数，建议范围 1-10000，数值越高越优先                            |
| **有效期**     | `Validity`        | 规则的生效时间区间   | `start` / `end` 时间戳对                                          |

### 7.2 决策值（四值）

| 中文术语   | 英文标识         | 定义                 | 处理建议                    |
| ---------- | ---------------- | -------------------- | --------------------------- |
| **允许**   | `allow`          | 显式授权通过         | 放行并执行 obligations      |
| **拒绝**   | `deny`           | 显式授权拒绝         | 阻断并记录审计              |
| **不适用** | `not_applicable` | 无匹配策略           | 按默认策略处理（通常 deny） |
| **不确定** | `indeterminate`  | 策略可匹配但求值失败 | 高敏场景按 deny 处理        |

### 7.3 合并算法

| 中文术语         | 英文标识                 | 语义                     | 适用场景                   |
| ---------------- | ------------------------ | ------------------------ | -------------------------- |
| **拒绝覆盖**     | `deny-overrides`         | 任一 deny 即最终 deny    | 默认算法，高敏域强制使用   |
| **允许覆盖**     | `permit-overrides`       | 任一 allow 即最终 allow  | 低风险公开域（高敏域禁用） |
| **首次适用**     | `first-applicable`       | 按优先级首个匹配即生效   | 需要明确顺序的场景         |
| **有序拒绝覆盖** | `ordered-deny-overrides` | 按优先级排序后 deny 优先 | 需要优先级裁决的场景       |

### 7.4 后置动作

| 中文术语 | 英文标识      | 定义                 | 强制性                     |
| -------- | ------------- | -------------------- | -------------------------- |
| **义务** | `Obligations` | 决策后必须执行的动作 | 强制执行，失败需回滚或告警 |
| **建议** | `Advice`      | 决策后建议执行的动作 | 非强制，用于提示或引导     |

## 8. 约束术语

| 中文术语     | 英文标识                   | 定义                       | 示例                                        |
| ------------ | -------------------------- | -------------------------- | ------------------------------------------- |
| **职责分离** | `SoD` (Separation of Duty) | 防止互斥动作由同一主体执行 | `approve_payment` 与 `execute_payment` 互斥 |
| **基数约束** | `Cardinality`              | 限制关键角色或关系的数量   | 租户 root 角色上限 2 人                     |

## 9. 一致性与质量术语

### 9.1 一致性级别

| 中文术语     | 英文标识            | 定义                   | 适用场景                             |
| ------------ | ------------------- | ---------------------- | ------------------------------------ |
| **强一致**   | `strong`            | 读取最新写入结果       | 高敏写操作、高敏读取                 |
| **有界过期** | `bounded_staleness` | 允许有限延迟（毫秒级） | 常规列表与检索场景                   |
| **最终一致** | `eventual`          | 允许短暂陈旧           | 低风险、可容忍延迟场景（高敏域禁用） |

### 9.2 属性质量

| 中文术语         | 英文标识     | 定义                 | 约束方式     |
| ---------------- | ------------ | -------------------- | ------------ |
| **来源可信度**   | `authority`  | 属性来源的可信级别   | 白名单机制   |
| **时效**         | `freshness`  | 属性的更新时效       | TTL 阈值     |
| **撤销传播时效** | `revocation` | 撤销后传播生效的时效 | 最大延迟阈值 |
| **可验证程度**   | `assurance`  | 属性可被验证的程度   | 验证机制配置 |

## 10. 入管与配置术语

### 10.1 入管模式

| 中文术语     | 英文标识          | 定义                                              | 适用场景         |
| ------------ | ----------------- | ------------------------------------------------- | ---------------- |
| **开放兼容** | `compat_open`     | 仅校验硬必填，缺失动态字段允许入管并告警          | 存量对象迁移初期 |
| **平衡兼容** | `compat_balanced` | 硬必填 + Profile 必填必须满足，条件必填可限时补录 | 渐进治理阶段     |
| **严格兼容** | `compat_strict`   | 所有必填全部满足否则拒绝                          | 高成熟度租户     |

### 10.2 必填类型

| 中文术语         | 英文标识              | 定义                   | 字段示例                                              |
| ---------------- | --------------------- | ---------------------- | ----------------------------------------------------- |
| **硬必填**       | `HardRequired`        | 平台统一的最小必填集合 | `tenant_id`、`object_id`、`object_type`、`created_by` |
| **Profile 必填** | `ProfileRequired`     | 按入管配置档决定的必填 | 按 Profile 定义                                       |
| **条件必填**     | `ConditionalRequired` | 按上下文条件追加的必填 | 高敏对象需 `data_domain`、`retention_class`           |

## 13. 使用规范

### 13.1 命名规范

1. **代码标识**：使用英文标识，采用 `snake_case`（配置文件、数据库）或 `camelCase`（API/代码）。
2. **文档表述**：首次出现使用"中文术语（英文标识）"格式，后续可单独使用中文术语。
3. **API 字段**：与本文档保持一致，禁止自造字段名。

### 13.2 禁止事项

1. 禁止用"用户"泛指 `Subject`，必须区分具体类型。
2. 禁止用"资源"混淆 `Object`，`Resource` 仅用于外部标准映射。
3. 禁止使用未在 `action_catalog` 中注册的动作。
4. 禁止使用未在对应关系域 catalog（`subject_relation_type_catalog` / `object_relation_type_catalog` / `subject_object_relation_type_catalog`）中注册的关系类型。
5. 禁止在高敏域使用 `eventual` 一致性或 `permit-overrides` 算法。

### 13.3 版本化

1. 本文档随设计演进同步更新。
2. 术语变更需同步更新所有相关文档、代码、API。
3. 新增术语需在本文档中注册后方可使用。
