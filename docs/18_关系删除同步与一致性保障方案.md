# 关系删除同步与一致性保障方案

> 文档编号：18  
> 更新日期：2026-03-11  
> 依赖主文档：`10_企业可配置权限模型开发设计.md`、`11_权限配置JSON_Schema草案.md`、`12_权限发布门禁规则样例.md`

## 1. 背景

外部系统在接入 ACL 控制面时，关系同步目前以 `upsert` 为主。  
当资源从一个工作组迁移到另一个工作组、或成员关系撤销时，如果控制面未同步删除关系边，会导致：

1. 旧关系残留，判权继续放行。  
2. `decision_search` 返回“多余对象”。  
3. 影子比对时出现“ACL 多返回”差异。

本方案明确“关系删除同步”的控制面保障与验证点，确保动态变更后权限收敛。

## 2. 目标

1. 控制面支持关系删除事件（`operation=delete`）并保证持久化一致。  
2. 删除后的关系对 `evaluate / search / inference` 立即生效。  
3. 提供可回归的测试验证删除语义。

## 3. 代码核查与修改点（ACL 项目）

### 3.1 控制面事件入口

现有实现已支持 `operation=delete`（需确认在当前版本仍然有效）：

- `apps/api/src/main.ts`  
  - `POST /control/relations:events`  
  - 处理逻辑：`operation === 'delete'` -> `persistence.deleteControlRelation(...)`

建议动作：

1. 确认删除分支保留且未被权限或网关拦截。  
2. 对删除事件也写入 `control.relation.synced` 审计（当前已包含 `deleted_count`，保持即可）。

### 3.2 持久化层删除

当前持久化层已具备删除实现（需确认 driver 覆盖完整）：

- `packages/persistence/src/postgres.ts`  
  - `deleteControlRelation(...)` 使用 `acl_validation_records` 删除。  
- `packages/persistence/src/memory.ts`  
  - `deleteControlRelation(...)` 删除内存关系记录。

建议动作：

1. 确认所有 persistence driver 都实现删除（含扩展存储）。  
2. 删除后查询 `listControlRelations` 不再返回旧记录。  

### 3.3 决策与推导一致性

关系推导依赖 `listControlRelations`：

- `apps/api/src/relation-inference.ts`

建议动作：

1. 删除关系后，推导链路应不可再命中旧关系。  
2. `decision_search` 与 `evaluate` 使用的控制面关系应与删除后的数据一致。

## 4. 接入侧（外部系统）行为约定

外部系统在更新关系时，建议采用“先删旧，再写新”的一致性策略：

1. 资源工作组迁移：  
   - 删除旧 `belongs_to`  
   - upsert 新 `belongs_to`
2. 成员关系撤销：  
   - 删除 `member_of / manages / read_only_of`
3. 资源归属变更：  
   - 删除旧 `owns`  
   - upsert 新 `owns`

删除事件建议带上 `scope`（若历史数据存在 `scope=default`），以避免 key 不匹配。

## 5. 测试建议（ACL 项目）

新增或补充以下用例：

1. 关系删除后 `listControlRelations` 不再返回该边。  
2. 关系删除后 `evaluate` 从 `allow` 变为 `deny`（或反向）。  
3. 资源迁移场景：  
   - 旧工作组用户不可访问  
   - 新工作组用户可访问  
4. 删除后 `decision_search` 不再包含旧对象。

以上用例建议覆盖 `memory` 与 `postgres` 持久化实现。
