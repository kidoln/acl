# 开发任务拆分与实施计划（可执行版）

> 文档编号：13
> 更新日期：2026-03-04
> 需求解释顺序：`docs/10` > `docs/11` > `docs/12` > `AGENTS.md`
> 术语遵循：[`00_术语统一规范.md`](./00_术语统一规范.md)

## 1. 文档目标

将权限模型设计文档转化为可执行的工程任务拆分，输出可直接进入迭代的工作包（Work Package, WP）、依赖关系、验收标准、里程碑与风险控制。

## 2. 需求基线与优先级

### 2.1 依据 `docs/10`（主设计，最高优先级）

优先落地以下主能力（按实施顺序）：

1. 配置元模型与统一文档结构（第 3、13.2 节）。
2. 决策语义与合并算法（第 7、10 节）。
3. 约束能力：SoD、Cardinality（第 8 节）。
4. 生命周期处理，尤其 `subject_removed` 默认语义（第 9 节）。
5. 五层校验、冲突检测、错误码与门禁（第 13.5~13.12 节）。
6. 控制面/数据面接入链路（第 10.4 节）。
7. 可视化治理能力（第 14 节）。

### 2.2 依据 `docs/11`（Schema 与基础校验）

形成以下强约束：

1. 顶层配置块必须齐全：`model_meta`、`catalogs`、`object_onboarding`、`relations`、`policies`、`constraints`、`lifecycle`、`consistency`、`quality_guardrails`。
2. `model_meta.status=published` 时，目录与规则必须非空。
3. `consistency.default_level=bounded_staleness` 时，`bounded_staleness_ms` 必填且 `>=1`。
4. 语义校验器必须补齐 Schema 外规则（选择器 AST、高敏域限制、关键生命周期处理器、冲突可消歧等）。

### 2.3 依据 `docs/12`（发布门禁）

形成发布流水线的分级门禁：

1. `P0` 阻断（不可豁免）。
2. `P1/P2` 复核（可豁免但必须留痕+到期失效）。
3. `P3` 提示（不阻断但生成治理工单）。
4. 基础版与强合规版采用不同阈值档位。

### 2.4 依据 `AGENTS.md`（工程落地约束）

工程默认约束：

1. 技术基线：`pnpm workspace + turbo`、`Node.js 20+`、`NestJS`、`Next.js`、`PostgreSQL + Redis`、`ajv`。
2. 文档/注释中文优先，代码标识英文。
3. 不执行任何写入 git 操作。
4. 不自动修改数据库 schema；涉及 schema 变更必须先获确认。

## 3. 范围与分阶段目标

### 3.1 MVP（阶段 1）

目标：上线“可发布、可判权、可回放”的最小闭环。

包含：

1. 配置 Schema 校验 + 语义校验 + 安全校验 + 可执行性校验（五层中 2~5 层）。
2. PDP 决策引擎（四值决策 + 合并算法 + obligations/advice）。
3. SoD/Cardinality 与高敏域限制。
4. 生命周期 `subject_removed` 关键处理链。
5. 发布门禁 `P0~P2`（至少基础版）。
6. 决策审计与回放 API（最小字段集）。

### 3.2 增强版（阶段 2）

目标：提升治理可用性与可维护性。

包含：

1. 门禁 `P3` + 治理工单联动。
2. 影响面模拟报告（14.7 模板最小版）。
3. 控制台矩阵视图、关系图视图、回放视图 MVP。
4. 外部应用接入 `bridge_standard`（对象/关系增量同步 + SCIM 同步预留）。

### 3.3 企业版（阶段 3）

目标：大规模接入与运营化。

包含：

1. `bridge_enterprise`（网关 ext_authz 联动、影子对比、灰度发布）。
2. 多租户治理模板与强合规模板。
3. 运行指标与 SLO 告警体系（门禁指标、鉴权质量指标、生命周期处理时效）。

## 4. 模块拆分（Monorepo）

建议目录（第一版）：

```text
apps/
  api/                 # NestJS：Control Plane + PDP API + Gate API
  console/             # Next.js：治理控制台
packages/
  schema/              # JSON Schema 与版本管理
  policy-dsl/          # 选择器解析、AST、类型检查
  validator/           # 五层校验器（结构/Schema/语义/安全/可执行）
  engine/              # PDP 决策引擎、合并算法、obligations
  lifecycle/           # 生命周期事件处理与重算
  gate/                # P0/P1/P2/P3 门禁执行器
  audit/               # 审计模型与回放查询
  shared-types/        # 跨端共享类型
  pep-sdk/             # 业务接入 SDK（可后置）
```

## 6. 迭代排期建议（8 周）

### 迭代 1（第 1~2 周）

1. 完成 WP-00 ~ WP-03。
2. 验收目标：配置可做结构化解析，选择器可输出 AST 与错误路径。

### 迭代 2（第 3~4 周）

1. 完成 WP-04 ~ WP-06。
2. 验收目标：五层校验闭环 + PDP 可稳定返回四值决策。

### 迭代 3（第 5~6 周）

1. 完成 WP-07 ~ WP-10。
2. 验收目标：生命周期关键事件可触发权限收敛；发布门禁可阻断/复核；决策可回放。

### 迭代 4（第 7~8 周）

1. 完成 WP-11 ~ WP-12。
2. 验收目标：控制台可查看矩阵与门禁结果；发布前模拟报告可出结论。

## 7. 验收标准（DoD）

每个工作包必须同时满足：

1. 功能验收：满足对应章节规则，不弱化 `docs/10` 平台底线。
2. 测试验收：单测覆盖主分支逻辑，关键链路有集成测试。
3. 可观察性验收：关键决策链有日志与指标。
4. 安全验收：高敏域限制（禁止 `eventual`、禁止弱化合并语义）可被自动校验。
5. 文档验收：API/配置字段与术语文档一致。

## 8. 测试拆分

| 层级 | 测试目标 | 关键用例 |
| --- | --- | --- |
| Unit | 模块内规则正确性 | 合并算法、SoD、Cardinality、错误码映射 |
| Integration | 跨模块链路正确性 | 配置发布 -> 门禁 -> 生效 -> 决策回放 |
| E2E | 控制台关键流程 | 创建模型、模拟发布、复核豁免、最终发布 |
| Regression | 历史行为不回退 | 决策结果快照对比、门禁阈值快照 |

## 9. 风险与缓解

| 风险 | 表现 | 缓解方案 |
| --- | --- | --- |
| 选择器 DSL 过于复杂 | AST 与运行时语义漂移 | 限制白名单函数，先实现最小 DSL 子集 |
| 规则冲突检测成本高 | 发布耗时过长 | 先做静态可消歧检测，复杂图检测异步化 |
| 生命周期重算影响面大 | 高峰期延迟上升 | 事件分片、异步队列、按租户限流 |
| obligations 执行依赖外部系统 | 判权链不稳定 | obligations 执行器做降级与补偿，但保留 P0 阻断语义 |

## 10. 文档差异处理记录（按 `docs/10` 优先）

当前识别到的差异与处理策略：

1. `ATTRIBUTE_SOURCE_UNTRUSTED` 在 `docs/10` 的错误码表偏阻断语义，但门禁分层又放在 `P1`。
处理：实施时以 `13.12` 分层门禁为主，在“高敏域/强合规 profile”中允许策略升级为 `P0`。

2. `object_onboarding.compatibility_mode` 在 `docs/10` 字段字典出现“可选”描述，而 `docs/11` 设为必填。
处理：按 `docs/10` 的统一结构与 `docs/11` 可执行 Schema，落地为“必填”。

> 说明：后续进入实际开发时，应在提交说明持续记录新增差异与处理依据。
