实现与讨论时，按以下顺序解释需求：

1. `docs/10_企业可配置权限模型设计.md`（主设计，优先级最高）
2. `docs/11_权限配置JSON_Schema草案.md`（配置结构与基础校验）
3. `docs/12_权限发布门禁规则样例.md`（发布门禁与阈值样例）

若文档间出现不一致，以 `docs/10` 为准，并在 PR/提交说明中记录差异处理。

## 1. TS 全栈技术路线（推荐基线）

目标：前后端统一 TypeScript，优先提升迭代速度与类型一致性。

1. Monorepo：`pnpm workspace` + `turbo`
2. Backend API：Node.js 20+ + `NestJS`（或 `Fastify` + `zod`）
3. Frontend Console：`Next.js` + React + TypeScript
4. DB：PostgreSQL（关系数据）+ Redis（缓存/短 TTL）
5. 配置校验：JSON Schema（Draft 2020-12）+ `ajv`
6. 类型共享：`zod` 或 schema-to-ts 生成共享类型
7. 测试：`vitest` + `supertest` + Playwright（控制台关键流程）

## 3. 文档语言与风格

1. 代码标识符使用英文。
2. 文档和注释优先中文，必要处补英文术语。

## 4. 基本规则

- 当前日期* 2026年3月，上网检索信息要用最新的日期，或者从配置中读取对应包版本再查询。
- 网络代理: 访问互联网信息失败时，尝试使用代理：`127.0.0.1:7897`
- 服务重启: 不要重启前后端服务，它们都是 reload 模式，自动生效的。改变配置确实需要重启的时候，让用户手动重启。
- 数据库: 不要自动修改数据库 schema，任何需要修改数据库 schema 的操作，必须停下来获得用户确认后，再进行修改。
- 禁止操作:
  - 禁止执行任何写入 git 的操作， 包括将代码移动到暂存区，也不允许，除非用户明确要求操作 git
- 代码分解: 要合理安排代码文件大小，避免出现单文件过大的情况
- 文档管理: 如果用户没有在对话中明确要求写文档，绝对不允许创建文档或更新文档。如果明确要求创建文档，新文档请写入 `docs/`，命名：`<编号>_<name>.md`, 顺序编号
- 绘图: 文档中绘图统一用 mermaid 绘制，注意使用商务配色，不要出现深色背景深色文字的情况。
